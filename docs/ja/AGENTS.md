# AIエージェントへの指示 - Lepus開発ガイドライン

`AGENTS.md`および`docs/`以下のマークダウンファイルは、AAIF（Agentic AI Foundation）のガイドラインに従って作成されています。

このリポジトリには、エンジニアリングチームのドキュメンテーション標準、レビューポリシー、およびPRテンプレートが含まれています。これは**ドキュメンテーション専用**であり、コードやビルド/テストの自動化は含まれていません。

## AIエージェント向けの主要な知識

### 1. リポジトリの目的と範囲

- **内部使用のみ**：Lepus Group外部からのPRは受け付けられません。
- すべてのコンテンツはドキュメンテーション用であり、コード実装用ではありません。
- 主要ファイル：`README.md`、`README.ja.md`、`CONTRIBUTING.md`、`docs/en/REVIEW_POLICY.md`、`docs/ja/REVIEW_POLICY.md`、`docs/en/`および`docs/ja/`のPRテンプレート。

### 2. プルリクエストのワークフロー

- 変更には**`main`からブランチを作成**します。
- PRテンプレート（`docs/en/PULL_REQUEST_TEMPLATE.md`または`docs/ja/PULL_REQUEST_TEMPLATE.md`）を使用します。
- PRには以下を含める必要があります：
  - 背景/目的（チケットリンク、根拠）
  - 主な変更点（箇条書き）
  - 受け入れ基準（チェックリスト）
  - 影響を受けるパッケージ/領域
  - テストの証拠（スクリーンショット、CIの結果、再現手順）
- **外部からのPRは不可**—適応させたい場合はフォークしてください。

### 3. レビューポリシーのハイライト

- **レビューレベル**：
  - L0：軽微な修正（簡単なチェック、CI/テストに重点）
  - L1：機能の追加/変更（ACと証拠が必要）
  - L2：高リスク（スキーマ/セキュリティ/外部統合）→ペアレビュー+手動検証
- **必須**：受け入れ基準、テスト結果、および再現手順。
- **コミット規則**：`test:`、`fix:`、`refactor:`のプレフィックスを使用します。大規模な名前変更/フォーマット変更→別のPRで。
- **明確な影響範囲**：影響を受ける領域（API/DB/認証/UI/外部システム）を常に記述します。

### 4. パターンと規約

- **単一責任の原則**：バグ修正とリファクタリングは別々のPRにすべきです（ただし、最小限の、振る舞いを保持するリファクタリングは除く）。
- **RACIマトリックス**：開発者（実装/テスト）、レビューア/テックリード（品質/設計/リスク）、QA/PM（ACの定義/検証）。
- **バグ修正のためのテストファースト**：再現テスト→修正→リグレッションチェック。

### 5. 言語とドキュメンテーション

- すべてのドキュメントは英語（`docs/en/`）と日本語（`docs/ja/`）の両方で利用可能です。
- PRとレビューには、適切な言語バージョンを参照してください。

## 例

- 詳細なレビュー基準とRACIの役割については、`docs/en/REVIEW_POLICY.md`および`docs/ja/REVIEW_POLICY.md`を参照してください。
- 必要なセクションとチェックリストについては、PRテンプレートを参照してください。
