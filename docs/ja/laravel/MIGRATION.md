# マイグレーション

このドキュメントでは、Laravel アプリケーションにおけるマイグレーションファイルの作成に関するガイドラインを定めます。

## 1. 安全性

- **[厳守]** `up()` で行った変更を完全にロールバックできる `down()` を必ず実装する。`down()` は `up()` と対称になるよう記述する。
- **[厳守]** カラム削除・テーブル削除を行う場合、対応するモデル・リレーションの変更を同一 PR に含める。
- **[厳守]** NOT NULL カラムを既存テーブルに追加する場合、DDL と DML を混在させないよう段階的なマイグレーションで対応する：（1）カラムを nullable または `DEFAULT` 値付きで追加する；（2）既存行のバックフィルは DDL と分離した別ステップで行う（小規模データは DML のみを含む専用マイグレーションファイル、大規模データは Seeder またはバッチジョブを使用）；（3）最後のマイグレーションで NOT NULL 制約を追加または DEFAULT を削除する。
- **[厳守]** `nullable()` の追加・削除など後方互換性に影響する変更は、デプロイ順序を考慮して段階的なマイグレーションに分割する。
- **[厳守]** 一度本番環境で実行されたマイグレーションファイルは編集しない。変更が必要な場合は新しいマイグレーションファイルを作成する。
- **[厳守]** `is_admin`、`is_active` などの権限・状態フラグカラムは `default(false)` / `default(0)` にする。デフォルトで権限を持たない設計にすることで、意図しない権限昇格を防ぐ。
- **[要修正]** DDL（スキーマ変更）と DML（データの挿入・更新・削除）を同一のマイグレーションファイルに混在させない。繰り返し実行するリファレンスデータは Seeder を使用する。スキーマ変更に伴う一回限りのバックフィルは、DML のみを含む専用マイグレーションファイルとして分離することを許容する（上記 NOT NULL カラムのルールを参照）。
- **[要修正]** 日時カラムは `timestamp()` ではなく `dateTime()` を使用する。`timestamp()` は MySQL では UTC 変換や 2038 年問題が生じるが、`dateTime()` は Laravel のスキーマビルダーが各 DB に適切な型へ変換するため影響を受けない。

## 2. パフォーマンス

- **[要修正]** 外部キー制約を追加する場合、対象カラムにインデックスを付与する。
- **[条件付き]** 大量データが存在するテーブルへの変更は、ロック時間を考慮した実装にする。

## 3. 保守性

- **[要修正]** `ENUM` 型の使用を避け、`string` または整数カラム＋定数で代替する。`ENUM` は値の追加・削除に ALTER TABLE が必要で、DB によっては再作成を伴い変更コストが高い。

## 4. 命名規則

- **[厳守]** テーブル名・カラム名はスネークケースで統一する（例：`user_profiles`、`created_at`）。
- **[厳守]** 過去のマイグレーションファイルの命名パターンと一致させる。
- **[厳守]** ファイル名は `YYYY_MM_DD_HHMMSS_<動詞>_<テーブル名>_table.php` の形式に従う（例：`2024_01_15_093000_create_users_table.php`）。
- **[厳守]** `create_` / `add_` / `remove_` / `rename_` / `drop_` など動作を表す動詞を先頭に付ける。
- **[条件付き]** 中間テーブルは単数形で命名し、「関係の主体となる抽象度の低い（より具体的なドメイン）エンティティ」のテーブル名を先に置く（例：`payments` と `http_communications` の中間テーブルは `payment_http_communication`）。なお、Laravel の `belongsToMany()` はデフォルトでテーブル名をアルファベット順に解決するため、このドメイン固有の命名規則に従う場合は第2引数にテーブル名を明示する必要がある（例：`belongsToMany(HttpCommunication::class, 'payment_http_communication')`）。

## 5. 冪等性

- **[厳守]** 既に存在するカラム・インデックスを再作成しようとしていないか確認する。

## 6. 移植性

- **[要修正]** 特定の DB ソフトウェアに依存する書き方は行わない。
